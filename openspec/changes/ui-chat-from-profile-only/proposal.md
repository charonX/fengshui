# UI 重构：聊天只能从档案进入

## 问题陈述

**当前问题**：
- 用户已经录入了档案，但进入聊天页面后，仍然需要重新输入生日等信息才能开始聊天
- 聊天页面 `/chat` 是独立入口，与档案系统脱节
- 每次聊天都需要重新建立上下文，体验割裂

**根本原因**：
- 设计初期，聊天和档案是两个独立的功能模块
- 没有建立"从档案进入聊天"的强制流程
- 聊天页面没有自动加载档案信息的机制

## 解决方案

**核心改变**：
1. **删除独立聊天页面** (`/chat`)，聊天功能只能从档案详情页进入
2. **档案详情页采用分栏布局**：左侧基础报告（只读），右侧 AI 聊天
3. **自动加载上下文**：进入档案详情页时，自动加载档案信息并开始聊天

## 变更范围

### 包含的内容
- 删除 `/chat` 独立页面及其相关入口
- 创建档案详情页 `/profiles/[id]`，包含左右分栏布局
- 左侧：基础报告（八字排盘、五行统计、大运走势、身强身弱、流年）
- 右侧：AI 聊天，自动加载档案信息到上下文
- 首页 (`/`) 简化，只保留档案管理入口

### 不包含的内容（后续迭代）
- 报告交互功能（点击报告内容触发特定问题）
- 多档案切换（在详情页切换到另一个档案）
- 聊天记录持久化（每次进入都是新对话）
- 报告编辑功能

## 预期效果

**用户体验改善**：
- 进入档案 → 自动看到报告 → 自动开始聊天，流程顺畅
- 不再需要重复输入生日信息
- 报告 + 聊天的并排布局，方便参考

**技术收益**：
- 简化的导航结构（更少的路由）
- 更清晰的页面职责（档案页 = 报告 + 聊天）

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 用户想不创建档案直接体验聊天 | 中 | 低 | 可以在首页保留"游客模式"入口（本次不包含） |
| 左侧报告占用空间，聊天区域太小 | 低 | 中 | 使用响应式布局，小屏幕可切换为标签页 |
| 自动加载失败时体验差 | 低 | 中 | 添加加载状态和错误处理 |

## 成功标准

- [ ] 用户可以从档案列表进入详情页
- [ ] 详情页左侧显示基础报告（由代码生成，非 AI）
- [ ] 详情页右侧可以正常聊天
- [ ] 聊天时自动包含档案信息（无需手动输入生日）
- [ ] 独立的 `/chat` 页面已删除
- [ ] 首页只保留档案管理入口
